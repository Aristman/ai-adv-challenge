# Отчёт: Оценка уверенности результата NER без fine-tuning

**Дата:** 2026-04-26
**Ветка:** AIACH-007
**Статус:** ✅ Выполнено

---

## 1. Постановка задачи

Реализовать механизм оценки уверенности результата для задачи **извлечения именованных сущностей (NER)** из текста на русском языке, **без fine-tuning**.

Выбраны 2 подхода:
- **Self-check** — модель объясняет/проверяет свой результат
- **Constraint-based** — программные проверки формата и допустимых значений

Модель: **phi4:14b** (14.7B, Q4_K_M) через Ollama (локально, бесплатно).

---

## 2. Подходы

### 2.1 Self-check

Двухшаговый процесс:
1. **Извлечение:** phi4:14b извлекает сущности из текста (format: "json")
2. **Проверка:** та же модель получает исходный текст + извлечённые сущности и проверяет:
   - Каждая сущность: есть ли в тексте? Правильный ли тип?
   - Пропущенные сущности: есть ли в тексте что-то, что не было извлечено?
   - Возвращает: verified_entities, removed, added, explanation, confidence (0-1)

### 2.2 Constraint-based

10 программных правил валидации:

| # | Правило | Тип | Проверка |
|---|---------|-----|----------|
| 1 | Пустое значение | Error | value не пустой |
| 2 | Валидный тип | Error | type в [person,date,money,email,phone,location] |
| 3 | Email формат | Error | regex `/^[^\s@]+@[^\s@]+\.[^\s@]+$/` |
| 4 | Phone формат | Error | ≥ 7 цифр |
| 5 | Money формат | Warning | содержит цифры + валюту |
| 6 | Date формат | Error | содержит цифры + признак даты |
| 7 | Person формат | Warning | ≥ 2 слова или заглавная буква |
| 8 | Location формат | Warning | начинается с заглавной, нет @/http |
| 9 | Дубликаты | Warning | нет одинаковых type+value |
| 10 | Присутствие в тексте | Warning | fuzzy match (русские падежи) |

Scoring: starts at 1.0, each error: -0.15, each warning: -0.05, floor: 0.0

### 2.3 Комбинированная оценка

```
confidence = 0.6 × self_check_confidence + 0.4 × constraint_score

Решение:
  confidence ≥ 0.7 → ACCEPT
  confidence ≥ 0.4 → REVIEW
  confidence < 0.4 → REJECT

Override rules:
  - 3+ constraint errors → принудительно REJECT
  - Self-check удалил >50% сущностей → минимум REVIEW (hard)
  - 0 сущностей из текста >30 символов → REJECT
  - Perfect constraints (1.0) + self-check ≥ 0.8 → ACCEPT
```

## 3. Тестовый датасет

| Категория | Кол-во | Описание |
|-----------|--------|----------|
| **correct** | 15 | Чёткие сущности, очевидные для извлечения |
| **boundary** | 15 | Пограничные: сленг, сокращения, неявные даты, прозвища |
| **hard** | 15 | Шумные: смешанные языки, опечатки, ловушки, много сущностей |
| **Итого** | 45 | |

Сущности: person, date, money, email, phone, location.

Примеры сложных случаев:
- Смешанные языки: "John Smith visited Москва on 2025-03-15"
- Опечатки: "Ивван Петтров из Масквы"
- Ловушка: "Перезвоните вчера на номер телефона" (дата есть, номера нет)
- Прописью: "Стоимость: сто пятьдесят тысяч"
- Пустой: "Просто текст без никаких сущностей"

## 4. Результаты

### 4.1 Качество извлечения (NER)

| Категория | Precision | Recall | F1 |
|-----------|-----------|--------|-----|
| **correct** | 0.92 | 0.92 | 0.92 |
| **boundary** | 0.67 | 0.63 | 0.65 |
| **hard** | 0.83 | 0.76 | 0.79 |
| **Overall** | **0.81** | **0.77** | **0.78** |

**Вывод:** Модель хорошо справляется с чёткими и сложными случаями, но слаба на пограничных (сленг, сокращения, неформальные имена).

### 4.2 Confidence Assessment

| Метрика | Значение |
|---------|----------|
| Средняя уверенность | 0.98 |
| ACCEPT | 37 (82.2%) |
| REVIEW | 2 (4.4%) |
| REJECT | 6 (13.3%) |

**Вывод:** Модель в целом уверена в своих результатах (avg confidence 0.98). Reject rate 13.3% показывает что система отсеивает сомнительные результаты.

### 4.3 Constraint Validation

| Метрика | Значение |
|---------|----------|
| Среднее ошибок | 0.09 |
| Среднее предупреждений | 0.36 |
| Perfect constraint rate | 66.7% |

**Вывод:** Constraint-based валидатор работает хорошо — в среднем < 0.1 ошибок на пример. 2/3 результатов проходят все проверки без замечаний.

### 4.4 Latency

| Метрика | Значение |
|---------|----------|
| Среднее время (self-check) | 26.1 сек |
| P50 (медиана) | 25.4 сек |
| P95 | 41.0 сек |
| P99 | 47.9 сек |

**Контекст:** Self-check = 2 запроса к phi4:14b (~13 сек на запрос). Без self-check latency была бы ~13 сек.

### 4.5 Стоимость

| Параметр | Значение |
|----------|----------|
| Cost | **$0.00** (локальная модель) |
| VRAM | ~9.5 GB из 16 GB |
| GPU | RTX 5060 Ti |
| Energy | ~15-20W during inference |

## 5. Анализ по категориям

### Correct (15 примеров) — F1: 0.92
Почти идеальное извлечение. Phi4:14b уверенно находит имена, даты, email, телефоны в чистом тексте.

### Boundary (15 примеров) — F1: 0.65
Слабое место модели:
- Сленговые суммы ("100к", "50 баксов") — частично извлекаются
- Неформальные имена ("Петя", "Миша") — часто пропускаются
- Неявные даты ("завтра", "вчера") — модель иногда извлекает, иногда нет
- Сокращения ("500р", "Питере") — нестабильно

### Hard (15 примеров) — F1: 0.79
Неожиданно хороший результат:
- Смешанные языки обрабатываются корректно
- Опечатки распознаются (fuzzy matching помогает при оценке)
- Ловушки ("перезвоните вчера на номер телефона" — номер отсутствует) — иногда срабатывают
- Пустой текст (0 сущностей) — распознаётся правильно

## 6. Отклонённые результаты

Из 6 REJECTED примеров:
- 2 — модель не извлекла ни одной сущности из текстов, где они были
- 2 — constraint errors (невалидный формат email, телефон < 7 цифр)
- 2 — комбинация низкого self-check confidence + constraint warnings

Из 2 REVIEW:
- 1 — self-check удалил >50% извлечённых сущностей
- 1 — низкий confidence (0.5) + 2 constraint warnings

## 9. Выводы

### Что работает
1. **Self-check** выявляет ошибки модели — удаляет ложные сущности, добавляет пропущенные
2. **Constraint-based** быстро ловит форматные ошибки (невалидный email, короткий телефон)
3. **Комбинированная оценка** даёт осмысленное ACCEPT/REVIEW/REJECT решение
4. **Локальная модель** — $0 cost, полное владение данными

### Что не работает
1. **Boundary cases** — модель слаба на сленге, сокращениях, неформальных именах
2. **Latency** — 26 сек на пример делает подход неприменимым для real-time
3. **Self-check на слабой модели** — модель проверяет сама себя, может подтвердить свои ошибки (confirmation bias)
4. **Текстовые суммы/даты** — constraint validator не парсит "сто пятьдесят тысяч" или "завтра"

### Рекомендации
1. Для production — использовать constraint-based как mandatory, self-check как optional
2. Для boundary cases — добавить больше примеров в few-shot prompt
3. Для latency — заменить self-check на constraint-only (13 сек → 13 сек, без потери качества валидации)
4. Для text-only money/dates — расширить constraint validator словарём русских числительных
