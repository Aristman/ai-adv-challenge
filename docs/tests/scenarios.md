# E2E Test Scenarios — Orchestrator with bug-fix & code-research

Ручные сценарии для end-to-end тестирования оркестратора с новыми типами агентов.
Запуск: `/orchestrator on` в pi, затем дать оркестратору задачу из сценария.

---

## Setup

1. Запусти pi в директории проекта
2. Подготовь фикстуры: `bash test-fixtures.sh setup all`
3. Проверь что баги на месте: `bash test-fixtures.sh verify`
4. Активируй оркестратор: `/orchestrator on`
5. Дай оркестратору задачу из сценария ниже
6. Наблюдай за поведением: какие воркеры спавнятся, в каком порядке, что возвращают

---

## Сценарий 1: Code Research — базовый

**Задача оркестратору:**
> Расскажи, как устроена навигация в этом проекте? Какие экраны есть и через что реализована навигация?

**Ожидаемое поведение:**
- [ ] Оркестратор спавнит `code-research` воркер (а не `explore`)
- [ ] Воркер получает tools: read, bash, grep, find, ls (БЕЗ write/edit)
- [ ] Воркер ищет файлы навигации (App.kt, Navigator, ScreenModel)
- [ ] Воркер читает build.gradle.kts для understanding зависимостей (Voyager)
- [ ] Воркер читает CLAUDE.md для контекста
- [ ] Ответ содержит: таблицу файлов, описание связей, выводы
- [ ] Воркер НЕ создаёт и НЕ модифицирует файлы
- [ ] Иконка воркера: 🔬

**Red flags:**
- ❌ Воркер пытается использовать `write` или `edit` — ошибка в tools
- ❌ Воркер создаёт файлы — нарушение read-only
- ❌ Оркестратор спавнит `explore` вместо `code-research` — wrong type

## Сценарий 2: Code Research — глубокий анализ

**Задача оркестратору:**
> Какой DI фреймворк используется в проекте? Где объявляются модули и как они связаны с presentation слоем?

**Ожидаемое поведение:**
- [ ] `code-research` воркер ищет Koin-модули (grep "module {", "singleOf", "factoryOf")
- [ ] Читает di/ директорию
- [ ] Трассирует зависимости: ViewModels ← UseCases ← Repositories
- [ ] Вывод содержит Mermaid-диаграмму связей
- [ ] Каждый вывод подкреплён file:line ссылкой
- [ ] Timeout не превышает 180s

**Red flags:**
- ❌ Нет file:line ссылок в выводе
- ❌ Воркер выдумывает код — нет подтверждения из исходников
- ❌ Timeout — воркер завис

## Сценарий 3: Bug Fix — ошибка компиляции

**Задача оркестратору:**
> При сборке проекта ошибка: "Unresolved reference: SomeClass" в файле presentation/viewmodel/SomeViewModel.kt на строке 15.

**Предварительная подготовка:**
```bash
bash test-fixtures.sh setup 3
```
Файл `composeApp/.../presentation/viewmodel/DiaryListViewModel.kt` — создан с отсутствующим импортом `DiaryEntry`.

**Ожидаемое поведение:**
- [ ] Оркестратор спавнит `bug-fix` воркер (а не `implement`)
- [ ] Воркер видит ошибку: `Unresolved reference 'DiaryEntry'` в `DiaryListViewModel.kt`
- [ ] Воркер получает tools: read, write, edit, bash, grep, find, ls
- [ ] **Фаза 1:** Воркер ищет затронутый файл и читает его
- [ ] **Фаза 2:** Воркер запускает `./gradlew build` и видит ошибку
- [ ] **Фаза 3:** Воркер трассирует причину (не-existent import, missing module dependency)
- [ ] **Фаза 4:** Воркер чинит минимальным edit (правит импорт или добавляет зависимость)
- [ ] **Фаза 5:** Воркер запускает `./gradlew build` снова — успешная сборка
- [ ] Воркер выдаёт Bug Fix Report по шаблону
- [ ] Иконка воркера: 🐛
- [ ] Write-lock: только один bug-fix/implement воркер одновременно

**Red flags:**
- ❌ Воркер не воспроизвёл баг перед фиксом (не запустил build)
- ❌ Воркер сделал рефакторинг вместо минимального фикса
- ❌ Воркер не запустил верификацию после фикса
- ❌ Воркер проигнорировал CLAUDE.md
- ❌ Оркестратор спавнит `implement` вместо `bug-fix`

## Сценарий 4: Bug Fix — падающий тест

**Задача оркестратору:**
> Тест GetDiaryEntriesUseCaseTest падает с ошибкой "Expected 2, but was 0" в методе `returns entries from repository`.

**Предварительная подготовка:**
```bash
bash test-fixtures.sh setup 4
```
Файл `shared/.../GetDiaryEntriesUseCaseTest.kt` — mock возвращает 1 entry, но тест утверждает `size == 2`.

**Ожидаемое поведение:**
- [ ] `bug-fix` воркер находит тестовый файл `GetDiaryEntriesUseCaseTest.kt`
- [ ] Воркер читает тест и понимает что он тестирует
- [ ] Воркер ищет тестируемый класс (GetDiaryEntriesUseCase)
- [ ] Воркер находит причину: mock не настроен, логика неверна, или тест некорректен
- [ ] Воркер чинит и запускает `./gradlew test` — тест проходит
- [ ] Воркер проверяет что другие тесты не сломались

**Red flags:**
- ❌ Воркер чинит симптом а не причину (например, меняет expected на actual)
- ❌ Воркер не запустил тесты после фикса
- ❌ Другие тесты начали падать

## Сценарий 5: Bug Fix — нарушение CLAUDE.md

**Задача оркестратору:**
> В файле presentation/viewmodel/DiaryListViewModel.kt есть публичный MutableStateFlow вместо StateFlow. Это нарушение правил проекта (см. CLAUDE.md).

**Предварительная подготовка:**
```bash
bash test-fixtures.sh setup 5
```
Файл `composeApp/.../presentation/viewmodel/EntryEditViewModel.kt` — публичный `MutableStateFlow` вместо `private val _state` + `asStateFlow()`.

**Ожидаемое поведение:**
- [ ] `bug-fix` воркер читает CLAUDE.md
- [ ] Воркер находит нарушение в указанном файле
- [ ] Воркер фиксит: `MutableStateFlow` → `private val _state`, `public val state = _state.asStateFlow()`
- [ ] Воркер проверяет что CLAUDE.md compliance чеклист пройден
- [ ] Фикс следует стилю проекта

**Red flags:**
- ❌ Воркер не прочитал CLAUDE.md
- ❌ Воркер оставил `public MutableStateFlow`
- ❌ Воркер использует `!!` в фиксе

## Сценарий 6: Параллельность — code-research + bug-fix

**Задача оркестратору:**
> Есть баг в навигации (SplashScreen не переходит на следующий экран). Сначала исследуй как устроена навигация, потом почини баг.

**Ожидаемое поведение:**
- [ ] Оркестратор декомпозирует на 2+ подзадачи
- [ ] **Первый воркер:** `code-research` — исследует навигацию
- [ ] Оркестратор синтезирует результат исследования
- [ ] **Второй воркер:** `bug-fix` — чинит на основе исследования
- [ ] Write-lock: code-research и bug-fix НЕ запускаются одновременно
  (bug-fix требует write slot, code-research — нет, но оркестратор
   может запустить code-research пока bug-fix не нужен)
- [ ] После bug-fix — верификация (`verify` воркер)

**Red flags:**
- ❌ Оркестратор запускает bug-fix ДО исследования (не знает где чинить)
- ❌ Два write воркера одновременно
- ❌ Нет verify после bug-fix

## Сценарий 7: Write-lock — два bug-fix не параллелятся

**Задача оркестратору:**
> Есть два бага: 1) Ошибка компиляции в файле A, 2) Падающий тест в файле B. Почини оба.

**Ожидаемое поведение:**
- [ ] Оркестратор создаёт 2 задачи с зависимостями
- [ ] Первый `bug-fix` воркер запускается
- [ ] Второй `bug-fix` воркер **ждёт** в очереди (write-lock)
- [ ] После завершения первого — второй стартует
- [ ] НЕ происходит одновременная запись в файлы

**Red flags:**
- ❌ Два bug-fix воркера работают одновременно
- ❌ Race condition на файлах

## Сценарий 8: Code Research — чтение из другой ветки

**Задача оркестратору:**
> Текущая ветка пустая. Расскажи что есть в ветке AIACH-001 — какие файлы, какая структура проекта.

**Ожидаемое поведение:**
- [ ] `code-research` воркер обнаруживает что текущая ветка пуста
- [ ] Воркер использует `git show AIACH-001:path` для чтения файлов из другой ветки

- [ ] Ответ содержит структуру проекта из AIACH-001

**Red flags:**
- ❌ Воркер говорит "нет файлов" и сдаётся
- ❌ Воркер пытается создать файлы

## Сценарий 9: Code Research — запрещённые команды

**Задача оркестратору (с наблюдением):**
> Исследуй как работает модуль X.

**Во время работы воркера проверь в логах:**
- [ ] Воркер НЕ вызывает `rm`, `mv`, `cp`, `mkdir`
- [ ] Воркер НЕ вызывает `git checkout`, `git stash`, `git commit`
- [ ] Воркер НЕ вызывает `npm install`, `gradle build` (это не его задача)
- [ ] Воркер использует только read-only команды: `grep`, `find`, `read`, `cat`, `git show`, `git log`

**Red flags:**
- ❌ Любая write команда в логах воркера

## Сценарий 10: Smoke — все 6 типов за одну сессию

**Задача оркестратору:**
> Проведи полный ревью проекта: 1) Исследуй архитектуру, 2) Найди потенциальные баги, 3) Если найдёшь — почини, 4) Проверь результат.

**Ожидаемое поведение:**
- [ ] `explore` или `code-research` — исследование архитектуры
- [ ] `plan` — планирование фиксов на основе исследования
- [ ] `bug-fix` или `implement` — исправление багов
- [ ] `verify` — верификация после фиксов
- [ ] Task list показывает все подзадачи
- [ ] Оркестратор выдаёт финальный отчёт

**Проверка в Task List:**
- [ ] Все задачи в статусе `completed`
- [ ] Видны owner IDs разных типов воркеров
- [ ] Зависимости между задачами корректны
